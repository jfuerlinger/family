# Internationalization (i18n)

FamilyHub ships in **German (default)** and **English**, powered by [next-intl v4](https://next-intl.dev/). All routes are locale-prefixed: `/de/todos`, `/en/calendar`, etc.

## How it is wired

Four pieces work together:

### 1. `src/i18n/routing.ts` — the single source of truth for locales

```ts
export const routing = defineRouting({
  locales: ['de', 'en'],
  defaultLocale: 'de',
});
```

Everything else (middleware, request config, zod validation of the user's locale preference in `src/lib/actions/settings.ts`) derives from this object.

### 2. `src/middleware.ts` — locale routing

A thin `createMiddleware(routing)` wrapper. It detects the locale, redirects `/` to `/de/...`, and rewrites `/{locale}/...` to the `app/[locale]/...` segment. The matcher excludes `/api`, `/_next`, `/_vercel`, and static files.

### 3. `messages/{locale}/{feature}.json` — per-feature message files

```
messages/
├── de/
│   ├── common.json     # app, common, nav, auth, onboarding, dashboard, settings
│   ├── todos.json      # "todos" namespace
│   ├── calendar.json   # "calendar" namespace
│   └── mindmaps.json   # "mindmaps" namespace
└── en/ (same structure)
```

Splitting by feature keeps diffs small and lets several people (or agents) work on different features without merge conflicts in one giant JSON file.

### 4. `src/i18n/request.ts` — merging at request time

For each request, the file imports all namespace files for the active locale and merges them into a single messages object:

```ts
const NAMESPACE_FILES = ['common', 'todos', 'calendar', 'mindmaps'] as const;
// ...
return { locale, messages: Object.assign({}, ...parts) };
```

Components then use the standard next-intl API:

```tsx
// Server component
const t = await getTranslations('dashboard');
// Client component
const t = useTranslations('todos');
```

### Navigation

Always import `Link`, `redirect`, `usePathname`, and `useRouter` from `src/i18n/navigation.ts` (not from `next/link` / `next/navigation`) inside pages and components — these variants keep the locale prefix intact automatically.

### Locale persistence

The user's preferred language is stored on `User.locale` and changed in **Settings**; the settings server action redirects to the same page under the new locale prefix.

## Adding a new string

1. Decide which namespace it belongs to: feature-specific → `todos.json` / `calendar.json` / `mindmaps.json`; shared UI, navigation, auth, dashboard, settings → `common.json`.
2. Add the key to **both** `messages/de/<file>.json` and `messages/en/<file>.json`. German first — it is the primary language.

   ```jsonc
   // messages/de/todos.json
   { "todos": { "emptyList": "Noch keine Aufgaben in dieser Liste." } }
   // messages/en/todos.json
   { "todos": { "emptyList": "No tasks in this list yet." } }
   ```

3. Use it: `t('emptyList')` with `useTranslations('todos')` / `getTranslations('todos')`.
4. For placeholders use ICU syntax: `"greeting": "Hallo, {name}!"` → `t('greeting', { name })`.

> **Tip:** Missing keys throw in development. If a page crashes with `MISSING_MESSAGE`, you forgot one of the two locale files.

## Adding a new locale (step by step)

Say you want to add French (`fr`):

1. **Declare the locale** in `src/i18n/routing.ts`:

   ```ts
   locales: ['de', 'en', 'fr'],
   ```

   This automatically updates the middleware and the zod validation of the settings form (both derive from `routing.locales`).

2. **Create the message files**: copy `messages/en/` to `messages/fr/` and translate all four files (`common.json`, `todos.json`, `calendar.json`, `mindmaps.json`). The request config imports `messages/${locale}/${name}.json` dynamically, so no code change is needed there.

3. **Expose it in Settings**: add a label for the new language to the `settings` section of every `common.json` (e.g. `"french": "Französisch"` / `"French"` / `"Français"`) and add the option to the language `<select>` in the settings form (`src/app/[locale]/(app)/settings/settings-forms.tsx`).

4. **Check date formatting**: anything using `date-fns` locales needs the matching `date-fns/locale` import where dates are formatted.

5. Run the app and click through every page in the new locale — missing messages fail loudly in dev.

## Translation conventions

- **German uses du-Form**, consistently and informally: „Melde dich bei deiner Familie an", „Teile diesen Code…" — never „Sie". FamilyHub is a family app; formal German would feel out of place.
- **Tone**: short, friendly, encouraging. Empty states may be playful („Keine offenen Aufgaben – stark!" / "No open tasks — nice!").
- **English** follows the German source in meaning, not word-for-word — write natural English, not translated German.
- **Punctuation**: German uses „…" quotes and ` – ` (en dash); English uses "…" and ` — ` (em dash). Use the real ellipsis character (`…`).
- **Keys** are camelCase and describe meaning, not position (`invalidInvite`, not `errorText3`).
- UI labels referenced in user documentation must match these message files **exactly** — when you rename a label, grep `docs/user/` for the old one.
