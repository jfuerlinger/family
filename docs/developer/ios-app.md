# iOS / iPad App (Capacitor)

FamilyHub ships a native iOS/iPad app built with [Capacitor](https://capacitorjs.com/). It is a thin native shell around the **hosted web app** — not an offline-capable native rewrite.

## How the wrapper works

Capacitor normally bundles web assets into the app. FamilyHub instead uses **server URL mode**: `capacitor.config.ts` sets `server.url` from the `CAP_SERVER_URL` environment variable, so the native WebView loads your hosted FamilyHub instance directly:

```ts
// capacitor.config.ts (simplified)
const config: CapacitorConfig = {
  appId: 'com.example.familyhub',
  appName: 'FamilyHub',
  webDir: 'public',
  server: process.env.CAP_SERVER_URL
    ? { url: process.env.CAP_SERVER_URL }
    : undefined,
};
```

Consequences:

- The app always shows the **live deployed version** — no App Store release needed for web updates.
- Login sessions are regular NextAuth cookies inside the WebView.
- The app is only as available as your server (see [Limitations](#limitations)).

## Prerequisites

- **macOS** with **Xcode** (latest stable) and its command-line tools
- **CocoaPods** (`sudo gem install cocoapods` or via Homebrew)
- Node.js + project dependencies installed (`npm install`)
- A **hosted, HTTPS-reachable FamilyHub instance** (see [deployment.md](deployment.md)) — for device testing, an `https://` URL is strongly recommended; iOS blocks plain-HTTP loads by default (App Transport Security)

## Build & run

1. **Point the app at your instance and sync:**

   ```bash
   CAP_SERVER_URL=https://family.example.com npm run ios:sync
   ```

   `ios:sync` builds the web assets and runs `cap sync ios`, regenerating the native project configuration (including the server URL) and installing pods.

2. **Open the Xcode project:**

   ```bash
   npm run ios:open
   ```

3. **Run on the Simulator:** select an iPhone/iPad simulator in Xcode's device dropdown and press **Run** (⌘R).

4. **Run on a real device:**
   1. Connect the device via USB and trust the computer.
   2. In Xcode → target **App** → *Signing & Capabilities*, select your Apple Developer team and let Xcode manage signing (a free Apple ID works for development builds; they expire after 7 days).
   3. Select the device and Run. On first launch, approve the developer certificate under iOS *Settings → General → VPN & Device Management*.

> **Tip:** When you change `CAP_SERVER_URL`, re-run `npm run ios:sync` — the URL is baked into the native config at sync time, not read at runtime.

## App Store notes

If you want to distribute beyond development builds:

- **Bundle ID & signing**: set a real bundle identifier (reverse-DNS you own) in Xcode and use a paid Apple Developer account with a distribution certificate/profile.
- **Icons & launch screen**: provide a full icon set in `ios/App/App/Assets.xcassets` (1024×1024 master required) and check the launch storyboard.
- **Versioning**: bump *Version* and *Build* in Xcode for every submission.
- **Review considerations**: pure "web wrapper" apps can be rejected under App Store guideline 4.2 (minimum functionality). Adding native touches (push notifications, share extensions, proper offline messaging) improves your odds. For private family use, **TestFlight** or an unlisted app may be the more pragmatic route.
- Archive via *Product → Archive* and upload through the Xcode Organizer.

## Limitations

- **Requires a reachable server.** The app is a WebView onto your hosted instance — no server, no app. There is no offline mode yet; without connectivity you'll see a connection error.
- **No native push notifications** yet.
- **iOS only** for now; an Android wrapper would follow the same pattern (`npx cap add android`).
- The `ios/` directory is generated by Capacitor; prefer making changes via `capacitor.config.ts` + `cap sync` rather than hand-editing the Xcode project where possible.
